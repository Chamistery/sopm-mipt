# ADR 0003 — Структура папок: feature folders, не layered

**Дата:** 2026-05-03
**Статус:** Принято

## Контекст

Когда фичи пишут параллельные агенты, главная боль — конфликт правок одних
и тех же файлов. Layered-структура (`components/`, `hooks/`, `services/`,
`pages/`) этого как раз провоцирует: добавление одной фичи требует правок
в 4 местах.

## Решение

Frontend разрезан по **фичам**, не по слоям:

```
src/
├── api/        # тонкий слой клиента + сгенерированный SDK
├── auth/       # auth store, RequireAuth, useCurrentUser — кросс-фичное
├── layout/     # AppShell, Sidebar — кросс-фичное
├── styles/     # tokens.css + global.css — кросс-фичное
├── features/
│   ├── auth/            # LoginPage
│   ├── profile/         # ProfilePage
│   ├── student-catalog/ # каталог проектов студента
│   ├── student-project/ # текущий проект студента/тимлида
│   ├── mentor-dashboard/# дашборд ментора
│   ├── coordinator/     # экраны координатора
│   └── errors/          # NotFoundPage, PlaceholderPage
├── router.tsx
└── main.tsx
```

Внутри каждой фичи собственные `components/`, `hooks/`, `*.test.tsx`,
`*.stories.tsx`. Public surface фичи описывается `index.ts`.

`api/`, `auth/`, `layout/`, `styles/` трогает **только тимлид-агент или
человек**, отдельным маленьким PR'ом. Это явно прописано в `CLAUDE.md`.

## Последствия

- 4–5 фич можно делать параллельно, в worktree-ветках, почти без конфликтов.
- Удалить фичу = удалить одну папку.
- Минус: дублирование некоторых утилит между фичами в начале. Это **ок** —
  выносить в `_shared/` начнём только когда увидим реальные 3+ повторений.

## Альтернативы

- _Layered (`components/`, `hooks/`, `services/`)._ Параллельная работа
  превращается в постоянные мерж-конфликты. Не подходит.
- _Atomic Design (`atoms/`, `molecules/`, `organisms/`)._ Хорошо для дизайн-
  систем, плохо для бизнес-фич — заставляет искусственно классифицировать
  компоненты.
