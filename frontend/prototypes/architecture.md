## 1. Обзор проекта

### 1.1 Что это
Веб-система для управления дисциплиной "Проектный практикум" в Высшей школе программной инженерии МФТИ. Автоматизирует весь цикл: подача заявок на проекты, распределение студентов, управление спринтами и задачами, отчётность, оценивание, защита.

### 1.2 Нормативная основа
Вся бизнес-логика определяется документом **"Положение о проектном практикуме ВШПИ"** (файл `Проект Положение ПП.docx`). Это фактическое ТЗ на систему.

### 1.3 Целевой стек
- **Frontend:** React (планируется)
- **Backend:** Go 1.24, goqu (SQL builder), pgx (PostgreSQL driver)
- **БД:** PostgreSQL 16
- **Деплой:** Docker, VDI-контейнер МФТИ (remote.vdi.mipt.ru)
- **Файловое хранилище:** Яндекс Диск API (библиотека `yadisk`)

### 1.4 Репозитории
- **Основной:** `git@github.com:Chamistery/sopm-mipt.git` (бэкенд + прототипы в `frontend/prototypes/`)
- **Зеркало прототипов:** `git@github.com:FGPy-cmpsc/Mock.git` (ветка `main`, папка `prototypes/`)

## 5. Бизнес-логика

### 5.1 Жизненный цикл проекта

```
Заявка (Черновик) → Утверждение (Координатор) → Опубликован → 
→ Распределение студентов → Активный → Спринты → Защита → Завершён
```

### 5.2 Распределение студентов

**Количество приоритетов** — задаётся переменной (сейчас = 5). Студент обязан указать все 5. После подачи приоритеты блокируются — изменить нельзя.

**Этапы:**

1. **Подача заявок.** До установленной даты студенты подают заявки, указывая 5 приоритетов. Студенты видят ВСЕ проекты, включая те, чьим требованиям не соответствуют (по курсу/баллу).

2. **Черновое распределение.** После дедлайна координатор запускает сервис распределения, который генерирует черновое распределение студентов по конкретным командам.

3. **Просмотр и редактирование ментором.** Ментор видит ВСЕХ студентов, подавших заявку на его проект (любым приоритетом). Отображение разбито на:
   - **Два блока:** подходящие по требованиям (курс + средний балл) и не подходящие
   - **Внутри каждого блока — 5 колонок** по приоритетам (1, 2, 3, 4, 5)
   - Ментор может перетаскивать студента из любой колонки в конкретную команду и обратно

4. **Приглашение студента.** Когда студент в списке команды (статус "Рекомендован"), ментор может нажать галочку → статус "Принято ментором", студент получает приглашение.

5. **Решение студента:**
   - **Принял** → статус "Принят", фиксируется в команде, все приглашения в другие проекты → "Авто-отклонено"
   - **Отказался** → статус "Студент отклонил", окончательно (повторное приглашение невозможно)
   - Студент может одновременно получить приглашения от нескольких менторов

6. **Исключение.** Координатор может исключить принятого студента из проекта → статус "Исключён"

7. После формирования команд ментор назначает тимлида

**ДКА статусов заявки:**
```
Ожидает → Не подходит (авто, по курсу/баллу)
Ожидает → Не рекомендован (сервис не поместил в команду)
Ожидает → Рекомендован (сервис поместил в команду)
Не подходит → Рекомендован (ментор перетащил)
Не рекомендован ↔ Рекомендован (ментор перетаскивает туда-обратно)
Рекомендован → Принято ментором (ментор нажал ✓)
Принято ментором → Принят (студент принял)
Принято ментором → Студент отклонил (окончательно)
Принято ментором → Авто-отклонено (принял другой проект)
Принят → Исключён (координатор исключил)
```

**Статусы видимые студенту** (только 4):
- **Рекомендован** (синий) — студент числится в команде
- **Не рекомендован** (серый) — студент не в команде
- **Принят** (зелёный) — студент зафиксирован в команде
- **Отклонено** (серый) — покрывает и ручной отказ, и авто-отклонение

Внутренние статусы, **не отображаемые студенту**: "Не подходит", "Ожидает".

### 5.3 Спринты и задачи

**Спринты** — свойство проекта. Настраиваются ментором при создании проекта:
- Количество, длительность (2-4 недели), даты
- У разных проектов разные спринты
- Все команды одного проекта на одной итерации

**Статусы задач:**

| Статус | Как устанавливается | Редактируемость |
|--------|--------------------|-|
| Ожидает аппрува | При создании (задача сразу ждёт подтверждения ментора). Нет статуса "Создана" — задача начинается здесь | Автор может редактировать. Автор может ОТМЕНИТЬ задачу (удалить) |
| Назначена | Ментор нажимает "Аппрувить" (ждёт наступления start_date) | Название, описание, часы — да. Даты — только ментор |
| Отклонена | Ментор нажимает "Отклонить" (задача исчезает с Ганта, не показывается пользователям) | Нередактируема |
| В работе | Автоматически при наступлении start_date (после аппрува) | Описание работы, MR — да. Даты — только ментор |
| На ревью | Студент нажимает "Отправить на ревью" | Нередактируема |
| Возвращена | Ментор возвращает с ревью (с комментарием) | Описание работы, MR — да |
| Готово | Ментор принимает задачу | Нередактируема |

**Цикл жизни задачи:**
```
                 [Студент отменил]
                       ↑
Ожидает аппрува → [Ментор] → Назначена → В работе → На ревью
                    ↓                                    ↓
                Отклонена                    [Ментор] Готово
                (скрыта)                        ↓
                                           Возвращена → В работе → ...
```

**Аппрув:** Задача создаётся сразу в статусе "Ожидает аппрува". Без аппрува ментора задача не может перейти в "Назначена" и далее "В работе".

**Отмена:** Студент/тимлид может отменить задачу, пока она в статусе "Ожидает аппрува". После аппрува задача становится постоянной и не может быть отменена.

**Ревью:** Проводит только ментор. Он может принять (→ Готово) или вернуть (→ Возвращена) с комментарием.

**Комментарии ментора** (`mentor_comments`):
- Массив объектов `{action, text}`, а не одиночная строка
- Действия: "Аппрув", "Отклонение", "Принятие", "Возврат"
- Пример: `[{"action":"Аппрув","text":"Задача утверждена."}, {"action":"Возврат","text":"Доработай описание."}]`
- Комментарий обязателен для Отклонения и Возврата, опционален для Аппрува и Принятия

**История событий** (`history`):
- Массив `{day, event}` для отслеживания дат ревью/возвратов/принятия на Ганте
- События: `review` (отправлена на ревью), `returned` (возвращена ментором), `accepted` (принята ментором)
- Отображаются как вертикальные цветные линии на барах Ганта (фиолетовый=review, жёлтый=returned, зелёный=accepted)

**Флаг просрочки** (`was_overdue`):
- Устанавливается автоматически если `end_date` прошла, а задача не на ревью/готово
- Сохраняется навсегда — даже если задача потом завершена
- Визуально: красная обводка статус-квадратика + бейдж "Просрочена" в попапе
- Просроченная задача отображается как "В работе" с красной обводкой (нет отдельного статуса "Истекла")

### 5.4 Отчётность

**Личные отчёты** — формируются из задач спринта:
- Каждый участник видит свои задачи из Ганта с описаниями и MR
- Все участники видят отчёты друг друга
- Отдельного текстового поля "Мой вклад" нет — отчёт = совокупность задач

**Командный отчёт** — заполняет тимлид:
- Три секции: "Что сделано", "Проблемы и риски", "План на следующий спринт"
- Два действия: "Сохранить черновик" и "Отправить на проверку"
- Ментор проверяет и оставляет комментарий (оценка команде НЕ ставится)
- Ментор оценивает каждого участника индивидуально (из 10) через таблицу `sprint_scores`

### 5.5 Итоговое оценивание

**Логика расчёта оценок пока не реализуется.** Формула и веса будут определены позже. Ментор оценивает каждого участника индивидуально (из 10 баллов за спринт) через таблицу `sprint_scores`.

### 5.6 Встречи

- Привязаны к команде и спринту
- Назначать может ментор или тимлид
- Поля: тема, дата, время, длительность, повестка, ссылка на конференцию, итоги
- **Подтверждение ментором:** если встречу создал тимлид — она требует подтверждения ментора (статус "Ожидает подтверждения"). Если создал ментор — сразу "Подтверждена"
- Статусы: "Ожидает подтверждения", "Подтверждена", "Отклонена", "Состоялась", "Отменена"
- Ментор может отклонить с указанием причины (`mentor_decline_reason`)
- Напоминание на дашборде: за неделю до конца спринта, если встреча не проводилась

## 6. UI/UX решения (зафиксированные)

### 6.1 Общие принципы
- Цветовая схема: глубокий индиго (`#1e2a5e`) + электрик-синий акцент (`#3b82f6`)
- Строгий/корпоративный стиль (МФТИ)
- Каждый элемент UI должен быть actionable — никаких декоративных метрик

### 6.2 Диаграмма Ганта
- Каждый спринт — отдельный блок со своей временной шкалой
- Текущий спринт сверху, завершённые ниже
- Sticky-колонки (Задача 220px, Статус-квадратик 12px, Часы 50px) не скроллятся горизонтально
- Статус отображается как цветной квадратик (12px) рядом с названием задачи (без текстовой пилюли, без заголовка колонки "Статус")
- Иконка ручки (карандаш) слева от названия своих задач, пустой спейсер для чужих
- История событий (`history`) отображается как вертикальные цветные линии на барах:
  - Фиолетовая линия = `review` (отправлена на ревью)
  - Жёлтая линия = `returned` (возвращена ментором)
  - Зелёная линия = `accepted` (принята ментором)
- Легенда цветов и линий отображается только на странице ментора
- Кнопка "Добавить задачу" внутри спринт-лейбла (только текущий спринт, только если canEdit)
- Вертикальная линия текущей даты (серая, от числа в кружке)
- Клик по своей задаче → попап редактирования, клик по чужой → readonly попап

### 6.3 Попап задачи (редактирование — студент/тимлид)
- Анимация: вылетает из названия задачи в Ганте (scale 0.05 → 1.0, transform-origin на элементе)
- Закрытие: по клику на оверлей, данные автосохраняются, попап улетает обратно
- Toast "Изменения сохранены" на 1.5 сек при наличии изменений
- Кнопка "Отправить на ревью" (фиолетовая) — только для задач в статусе "В работе"
- Кнопка "Отменить задачу" — только для задач в статусе "Ожидает аппрува" (до аппрува ментора)
- Кнопка "Создать задачу" (синяя) — для новых задач
- Для readonly/завершённых — кнопок нет
- Статус-select скрыт (статусы меняются автоматически/через кнопку ревью)
- Даты readonly для студента/тимлида (меняет только ментор)

### 6.3.1 Попап задачи (ментор)
- При клике на задачу ментор видит readonly-попап с кнопками действий:
- **"Ожидает аппрува":** две кнопки — "Аппрувить" / "Отклонить"
  - Клик на кнопку: выбранная заливается цветом, невыбранная остаётся outline
  - Появляется textarea для комментария с контекстным placeholder
  - Кнопка "Подтвердить" для отправки действия
  - Комментарий обязателен для "Отклонить", опционален для "Аппрувить"
- **"На ревью":** две кнопки — "Принять" / "Вернуть"
  - Аналогичная механика: выбор кнопки → textarea → "Подтвердить"
  - Комментарий обязателен для "Вернуть", опционален для "Принять"
- Для задач в других статусах — только просмотр (без кнопок действий)

### 6.4 Readonly попап (чужие задачи)
- Отдельный дизайн (не disabled-форма): карточка с заголовком, статус-бейджем, аватаром
- Секции: сроки, часы, описание, выполненная работа, MR
- Закрывается по крестику или клику на оверлей

### 6.5 Каталог проектов (нераспределённый студент)
- Карточки при наведении расширяются поверх соседей
- Выбранные — синяя рамка + бейдж "Выбран" (без затемнения!)
- Анимация "улетания" карточки во вкладку "Мои выборы"
- 5 слотов приоритетов расположены горизонтально (слева направо). Все 5 обязательны
- Drag-and-drop между слотами с FLIP-анимацией
- Подробная информация о проекте открывается inline

### 6.6 Отчёты
- Единый блок "Личные отчёты" для всех ролей
- Свой отчёт первый, чужие ниже
- Каждый отчёт = список задач из Ганта (кликабельные, открывают попап)
- Командный отчёт (только у тимлида) — отдельный блок сверху
- Прошлые спринты — раскрывающийся блок со всеми отчётами + комментарий ментора

### 6.7 Профиль студента
- Основная информация (readonly из МФТИ): ФИО, курс, группа, средний балл
- Контакты (редактируемые): email, Telegram, телефон
- Навыки (теги), ссылки (GitHub/GitLab/Портфолио — без LinkedIn), файлы (PDF/DOCX, без изображений)

### 6.8 Панель координатора (бывш. админ-панель)
- Дашборд: статистика + проекты (фильтр + поиск + inline-детали)
- Распределение: таблица заявок (5 приоритетов) + формирование команд с drag-and-drop
- Оценивание: таблица оценок + защиты (формула расчёта пока не реализуется)

## 7. Переключатель ролей (тестовый режим)

Для демонстрации в прототипах добавлен `<select>` в сайдбаре. При смене роли браузер переходит на соответствующий HTML-файл:

```javascript
const ROLE_PAGES = {
  student: 'student.html',
  student_assigned: 'student_assigned.html',
  teamlead_assigned: 'teamlead_assigned.html',
  mentor: 'mentor.html',
  admin: 'admin.html'   // координатор
};
```

В продакшене роль определяется из БД при авторизации. Роль `admin` в коде = Координатор.
